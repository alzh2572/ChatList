# План реализации ChatList

Поэтапный порядок работ по реализации программы на основании [PROJECT.md](PROJECT.md).

---

## Этап 1. Подготовка проекта

1. Обновить `requirements.txt`: PyQt6, httpx (или requests), python-dotenv.
2. Создать структуру модулей:
   - `main.py` — точка входа и GUI
   - `db.py` — работа с SQLite
   - `models.py` — логика моделей нейросетей
   - `network.py` — HTTP-запросы к API
3. Добавить `.env.example` с примерами переменных для API-ключей.
4. Убедиться, что `.env` указан в `.gitignore`.

---

## Этап 2. База данных (`db.py`)

1. Реализовать инициализацию БД: создание файла SQLite и таблиц по схеме из [DATABASE.md](DATABASE.md).
2. Добавить CRUD-функции для таблицы `prompts` (создание, чтение, поиск, удаление).
3. Добавить CRUD-функции для таблицы `models` (создание, чтение, обновление, удаление, фильтр активных).
4. Добавить CRUD-функции для таблицы `results` (сохранение выбранных строк, чтение истории).
5. Добавить функции для таблицы `settings` (получение и запись настроек по ключу).
6. Написать начальные данные (seed): одна-две тестовые модели, если таблица пуста.

---

## Этап 3. Работа с API (`network.py`, `models.py`)

1. В `models.py` описать структуру данных модели (имя, URL, имя переменной ключа, активность).
2. Реализовать загрузку API-ключей из `.env` по имени переменной из поля `api_id`.
3. В `network.py` реализовать базовую функцию отправки промта в одну модель (POST-запрос, обработка ошибок, таймаут).
4. Добавить поддержку разных типов API (OpenAI-совместимый формат как базовый; при необходимости — адаптеры для DeepSeek, Groq и др.).
5. Реализовать параллельную или последовательную отправку промта во все активные модели.
6. Вернуть список результатов в едином формате: `{model_name, response_text, error?}`.

---

## Этап 4. Временная таблица результатов (в памяти)

1. Создать класс или структуру для хранения текущей сессии запроса (промт + список ответов + флаг `selected`).
2. Реализовать очистку временной таблицы при новом промте.
3. Реализовать получение строк с `selected = True` для сохранения.

---

## Этап 5. Графический интерфейс (`main.py`)

### 5.1. Главное окно

1. Разделить окно на зоны: ввод промта, таблица результатов, панель действий.
2. Добавить поле ввода промта и кнопку «Отправить».
3. Добавить выпадающий список или боковую панель для выбора сохранённых промтов из БД.

### 5.2. Таблица результатов

1. Отобразить временную таблицу: колонки «Модель», «Ответ», «Выбрать» (чекбокс).
2. Заполнять таблицу после получения ответов от всех активных моделей.
3. Показывать индикатор загрузки на время запросов.

### 5.3. Действия пользователя

1. Кнопка «Сохранить» — записать отмеченные строки в `results`, очистить временную таблицу.
2. Кнопка «Новый запрос» — очистить поле промта и временную таблицу.
3. Обработка ошибок сети: вывод сообщения пользователю, строка с текстом ошибки в таблице.

---

## Этап 6. Управление моделями и промтами (GUI)

1. Добавить окно или вкладку «Модели»: список, добавление, редактирование, включение/отключение (`is_active`).
2. Добавить окно или вкладку «Промты»: просмотр сохранённых промтов, теги, выбор для повторной отправки.
3. Добавить окно «История результатов»: просмотр сохранённых записей из таблицы `results`.

---

## Этап 7. Настройки

1. Реализовать чтение/запись настроек через таблицу `settings` (путь к БД, таймаут запросов, тема и т.д.).
2. Добавить экран настроек в GUI (минимальный набор параметров).

---

## Этап 8. Дополнительные функции (опционально)

1. Поиск и сортировка во всех таблицах GUI.
2. Экспорт выбранных результатов в Markdown и JSON.
3. Логирование запросов в файл (`logs/`).
4. Сборка исполняемого файла через PyInstaller.

---

## Этап 9. Тестирование и финализация

1. Проверить полный сценарий: ввод промта → отправка → выбор строк → сохранение → просмотр истории.
2. Проверить работу с неактивными моделями и отсутствующими API-ключами.
3. Проверить повторный выбор сохранённого промта.
4. Обновить `README.md`: описание установки, настройки `.env`, запуск.

## План реализации: AI-ассистент для улучшения промтов

Поэтапный порядок работ по реализации функции из [PROJECT.md](PROJECT.md), раздел 7.

**Цель:** помочь пользователю улучшить промт перед отправкой в несколько моделей — через один запрос к выбранной модели OpenRouter.

**Зависимости:** этапы 1–5 (готовы), модуль `network.py`, вкладка «Запрос» в `main.py`, таблица `settings`.

---

### Этап A. Логика ассистента (`prompt_assistant.py`)

1. Создать модуль `prompt_assistant.py` — отдельно от массовой отправки промтов.
2. Описать структуру результата:
   - `original` — исходный текст;
   - `improved` — улучшенная версия;
   - `alternatives` — список из 2–3 переформулировок;
   - `adaptations` — словарь адаптаций по типам (`code`, `analysis`, `creative`) — опционально.
3. Сформировать системный промт ассистента: попросить модель вернуть ответ в **структурированном JSON**, чтобы его было легко разобрать.
4. Реализовать функцию `improve_prompt(text: str, model: AIModel) -> PromptImprovementResult`:
   - вызов существующей `send_prompt_to_model()` из `network.py`;
   - разбор JSON из ответа;
   - обработка ошибок (невалидный JSON, пустой ответ, сетевые ошибки).
5. Добавить fallback: если JSON не распарсился — показать сырой текст ответа как «улучшенный» вариант.

**Пример ожидаемого JSON от модели:**

```json
{
  "improved": "Улучшенный промт...",
  "alternatives": ["Вариант 1...", "Вариант 2...", "Вариант 3..."],
  "adaptations": {
    "code": "Промт для задач по коду...",
    "analysis": "Промт для анализа...",
    "creative": "Промт для креатива..."
  }
}
```

---

### Этап B. Настройки ассистента (`db.py`, `gui_tabs.py`)

1. Добавить в таблицу `settings` ключи:
   - `assistant_model_id` — ID модели из таблицы `models` для улучшения промтов;
   - `assistant_enabled` — включён ли ассистент (опционально).
2. На вкладке **Настройки** добавить выпадающий список «Модель для улучшения промта» — выбор из активных моделей OpenRouter.
3. Сохранять выбор в `settings` при нажатии «Сохранить настройки».
4. Значение по умолчанию: первая активная модель из `models` (например, `poolside/laguna-xs-2.1:free`).

---

### Этап C. Фоновый запрос (`main.py`)

1. Создать `ImprovePromptWorker(QThread)` по аналогии с `RequestWorker`:
   - принимает исходный текст промта и выбранную модель;
   - вызывает `improve_prompt()`;
   - сигналы `finished(result)` и `failed(message)`.
2. На время запроса блокировать кнопку «Улучшить промт» и показывать индикатор загрузки.
3. Логировать запросы ассистента через `log_utils.py`.

---

### Этап D. Интерфейс (`main.py` или `prompt_assistant_dialog.py`)

#### D.1. Кнопка на вкладке «Запрос»

1. Добавить кнопку **«Улучшить промт»** рядом с полем ввода (рядом с «Отправить»).
2. Кнопка активна, если поле промта не пустое и в настройках выбрана модель ассистента.

#### D.2. Окно результатов

1. Создать диалог `PromptAssistantDialog` (отдельный файл `prompt_assistant_dialog.py`):
   - блок **«Исходный промт»** — только чтение;
   - блок **«Улучшенный промт»** — основной результат;
   - список **«Альтернативы»** (2–3 варианта) — радиокнопки или список с выбором;
   - вкладка или сворачиваемая секция **«Адаптации»** — варианты для кода / анализа / креатива (если модель их вернула).
2. Кнопки внизу диалога:
   - **«Подставить в поле ввода»** — заменить текст в `prompt_input` выбранным вариантом;
   - **«Копировать»** — копировать выбранный вариант в буфер обмена;
   - **«Закрыть»**.
3. По умолчанию выбран «улучшенный» вариант.

#### D.3. Поведение после подстановки

1. После «Подставить в поле ввода» — закрыть диалог, текст появляется в поле промта.
2. Пользователь может сразу нажать «Отправить» для сравнения ответов моделей.

---

### Этап E. Обработка ошибок и граничные случаи

1. Пустой промт — предупреждение, запрос не отправляется.
2. Не выбрана модель ассистента в настройках — подсказка перейти на вкладку «Настройки».
3. Ошибка API / таймаут — сообщение в диалоге, исходный промт не изменяется.
4. Модель вернула меньше 2 альтернатив — показать то, что есть; пустые варианты скрыть.
5. Отсутствует блок `adaptations` — секцию адаптаций не показывать.

---

### Этап F. Тестирование

1. Ввести короткий промт → «Улучшить промт» → проверить отображение всех блоков в диалоге.
2. Выбрать альтернативу → «Подставить в поле ввода» → убедиться, что текст подставился.
3. Проверить работу при отсутствии API-ключа и при недоступной модели.
4. Проверить сохранение и загрузку `assistant_model_id` из `settings`.
5. Убедиться, что улучшение промта **не отправляет** запрос во все модели — только один вызов к модели-ассистенту.

---

### Этап G. Документация

1. Обновить `README.md`: описание кнопки «Улучшить промт» и настройки модели ассистента.
2. При необходимости дополнить `PROJECT.md` ссылкой на модуль `prompt_assistant.py`.

---

Этап A (логика) → Этап B (настройки) → Этап C (worker) → Этап D (UI) → Этап E → Этап F → Этап G
